Epic FHIR Integration

Production Integration Documentation

Deployment-facing technical reference for the Damaros Epic SMART on FHIR integration. Covers authentication, authorized services, PHI handling, screening determinism, access control, resiliency, and production controls.

01_PLATFORM_OVERVIEW

Platform Overview

Damaros is a clinical trial execution platform that performs deterministic, protocol-driven eligibility evaluation against Electronic Health Record data.

The system converts trial protocols into structured evaluation logic, screens approved cohorts against inclusion and exclusion criteria, and produces criterion-level evidence for coordinator review, institutional oversight, and audit reconstruction.

The Epic integration operates as a SMART on FHIR Backend Services application. Access is read-only by default. Optional write-back capabilities are disabled unless explicitly enabled after site authorization and deployment-level configuration.

02B_AUTHENTICATION_FLOW

Authentication Flow

Damaros uses SMART on FHIR Backend Services with JWT client assertions (RFC 7523). The client authenticates using a signed JWT presented to the Epic token endpoint. Access tokens are short-lived and held only in process memory for the active execution context.

GRANT_TYPEclient_credentials
ASSERTION_TYPEurn:ietf:params:oauth:client-assertion-type:jwt-bearer
TOKEN_ENDPOINTEpic-provided authorization server endpoint (DAMAROS_EPIC_TOKEN_URL)
JWT_ISS / JWT_SUBEpic-issued client ID for the active environment
JWT_AUDEpic token endpoint URL
JWT_EXPCurrent time + 300 seconds (capped)
JWT_JTIUUID generated per assertion
TOKEN_STORAGEIn-process memory cache only; not written to disk, database, or external store
Token acquisition uses bounded retry with exponential backoff. Authentication failures are surfaced as operator-visible error states and do not silently degrade to unauthenticated access.
02C_OAUTH_SCOPE

OAuth Scope

The configured scope is system/*.read, covering read access to Epic FHIR resources. No write scopes are included in the standard scope configuration. The resource families accessed under this scope are listed in section 03.

04_PHI_BOUNDARY

PHI Boundary and Data Flow

  • Cohorts are defined by approved protocol logic or institutional configuration.
  • Patient data is retrieved through FHIR read APIs or Bulk FHIR export depending on cohort size and site policy.
  • PHI is processed inside the institution-approved deployment boundary.
  • Patient-level clinical data is blocked from entering external LLM API calls by enforced classification gates at the application layer. Protocol parsing and non-PHI configuration tasks may use cloud-hosted inference; patient records do not pass through those paths.
  • Eligibility is computed deterministically against protocol inclusion and exclusion criteria.
  • Results are surfaced to authorized clinical research users with criterion-level evidence.
  • Audit artifacts preserve source FHIR resource references for review and reconstruction.
  • Data handling outside the deployment boundary is governed by executed data-use terms and institutional policy.
05_AUDIT_AND_REPLAY

Audit, Determinism, and Replay

  • Eligibility decisions are deterministic and reproducible for a fixed protocol version, cohort definition, evidence snapshot, and evaluation_as_of timestamp.
  • Every patient decision is recorded at criterion level with PASS, FAIL, or REVIEW status and associated rationale.
  • Evidence links back to source FHIR resource references where available.
  • Protocol amendments generate versioned re-evaluation outputs linked to a protocol_version_id and engine_version.
  • Replay uses the same protocol version, evaluation_as_of timestamp, normalized evidence snapshot, and deterministic ordering rules. Outcomes are classified as replay_ok, replay_diverged, or replay_blocked to surface any deviation from the original run.
  • No opaque decisioning is used for clinical eligibility outputs.
  • Audit logs are institution-scoped and access-controlled.
05A_ACCESS_CONTROL

Identity and Access Control

  • User access is role-scoped. Defined roles include coordinator, investigator, sub-investigator, research nurse, data manager, IRB reviewer, DSMB member, monitor, sponsor, site administrator, organization administrator, and related operator roles.
  • Each principal carries a subject identifier, role, organization scope, and authorized trial scope. Users access only studies, cohorts, evidence, and audit records within their authorized scope.
  • Administrative actions, export operations, and write-capable configuration changes are restricted to authorized roles.
  • Web sessions are issued with a fixed TTL; token expiry enforces time-bounded access.
  • Access events are logged for security review and operational audit.
05B_WRITE_BACK

Write-Back Controls

  • Write-back is disabled by default in all deployment configurations.
  • Write-back requires explicit site approval and must be enabled by setting DAMAROS_FHIR_WRITE_ENABLED=1 at deployment time. Attempting a write operation without this flag raises a runtime error with an explicit configuration message.
  • Permitted write-back is limited to institution-approved resource types such as consent or screening-status observations.
  • Standard eligibility screening does not require write-back to Epic.
06_PERFORMANCE

Performance and API Usage

  • FHIR API usage is designed to follow Epic operational performance guidance, targeting non-disruptive load on the Epic server.
  • Bulk FHIR export is the preferred mechanism for large cohort workflows, reducing per-patient query volume relative to individual FHIR reads.
  • Per-host rate limiting and bounded exponential backoff are applied to all FHIR requests on 429 and 5xx responses.
  • Bulk export polling uses a configurable interval (default 2 seconds) with a bounded maximum polling window.
  • Point queries use per-request retry caps to prevent unbounded retry behavior.
06A_ERROR_HANDLING

Error Handling and Resiliency

  • Token acquisition failures use bounded retry with exponential backoff and surface as operator-visible failure states.
  • Transient FHIR errors (429, 5xx) trigger exponential backoff with per-request retry caps.
  • Individual resource fetch failures are isolated; a failure on one resource type does not block retrieval of other resource types in the same patient bundle.
  • Missing or insufficient data is surfaced as REVIEW status rather than silently converting to eligibility PASS.
  • Failed ingestion, screening, and replay jobs preserve diagnostic metadata for authorized operators.
  • Retry operations are idempotent by design; no duplicate write behavior occurs when write-back is disabled.
07_SECURITY_AND_PRIVACY

Security and Privacy

  • Private signing keys are loaded from a configured secret store or secure file path and are never committed to source control.
  • Access tokens are short-lived and held only in process memory; they are not persisted to disk or database.
  • All FHIR API traffic is encrypted in transit (https:// enforced at the connector layer).
  • Application logging is designed to redact PHI patterns using a structured log processor active across all log output paths.
  • Data at rest encryption is deployment- and site-configured.
  • Clinical data is used only for approved research screening workflows.
  • Data is not sold, licensed, used for advertising, or used to train commercial AI models.
  • Retention, deletion, export handling, and purge behavior are site-configured and governed by executed data-use agreements and institutional policy.
