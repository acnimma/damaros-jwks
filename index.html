Epic FHIR Integration

Platform Documentation

Technical reference for the Damaros Epic SMART on FHIR integration, authentication model, data access scope, and privacy practices.

01_OVERVIEW

Platform Overview

Damaros is a deterministic, protocol-driven clinical trial screening platform that integrates with Epic Electronic Health Record systems to automate patient eligibility assessment with full audit lineage.

The Epic integration operates as a Backend Systems application using SMART on FHIR Backend Services authentication. All access is read-scoped by default. Optional write-back features (Observation, Subscription) remain disabled unless explicitly authorized by the institution after review.

Eligibility decisions are reproducible, criterion-traceable, and logged to institutional audit systems for compliance and transparency.

02A_ENVIRONMENT_SEPARATION

Environment Separation

Damaros maintains strict separation between non-production and production environments. Client IDs, key material, and FHIR endpoints are environment-scoped.

NON-PRODUCTIONSandbox integration, test data only. Client ID and keys are unique to the Sandbox environment. Key material is rotated monthly.
PRODUCTIONLive integration with institutional data. Client ID and keys are unique to Production. Key material is rotated quarterly or upon revocation request.
CROSS-ENV ISOLATIONProduction keys cannot authenticate against Sandbox and vice versa. Environment is enforced at the FHIR endpoint.
KEY ROTATIONPublic keys are available at JWKS endpoints above. Key rotation is communicated 30 days in advance. Legacy keys are valid for 60 days post-rotation.

03A_DATA_FLOW

Data Flow and Processing

1. Cohort Definition: Patient cohorts are defined by site protocol or institutional research criteria. Cohort definitions are configured prior to execution and immutable during a screening run.

2. Patient Data Retrieval: Data is retrieved via FHIR search endpoints or Bulk FHIR Export (preferred for large cohorts ≥1000 patients). All requests include Bearer token authentication and are subject to Epic ODB rate limits.

3. In-Memory Processing: Clinical data is processed within a secure, ephemeral execution environment. No intermediate data is persisted to application storage during evaluation.

4. Deterministic Eligibility Computation: Eligibility is computed deterministically per protocol criteria. Each decision is reproducible given identical input data and protocol version.

5. Results Surfacing: Eligibility results are surfaced only to authorized end-users via institution-scoped access. Screening status and metadata are retained for audit purposes per site agreement.

6. Data Boundary Compliance: Clinical data remains within Epic infrastructure boundaries. No data is transmitted to external systems except as explicitly authorized by institutional agreement.

04_SECURITY_AND_COMPLIANCE

Security and Compliance

Authentication & Authorization:

  • JWT client assertions are signed with RS384 and verified at the token endpoint.
  • Separate cryptographic key pairs are maintained for non-production and production environments.
  • Private keys are stored in a hardware-backed secrets manager and never committed to source control.
  • Tokens are short-lived (300 seconds) and never persisted to disk or logs.
  • Each institutional integration uses scoped OAuth credentials and cannot access other institutions' data.

Data Protection:

  • All API traffic is encrypted in transit using TLS 1.2 or higher.
  • No PHI is written to application logs, error messages, or debug output.
  • Clinical data is encrypted at rest in all storage layers.
  • Data is processed within institution-scoped boundaries; cross-institution access is cryptographically prevented.
04A_AUDIT_AND_TRACEABILITY

Audit and Traceability

Damaros maintains deterministic audit lineage for all eligibility decisions. No black-box decisioning is used in clinical outputs.

Eligibility Audit Record:

  • Decision Reproducibility: Each eligibility decision is reproducible. Given identical input data and protocol version, the same decision is reached.
  • Criterion-Level Evaluation: Each eligibility criterion is logged with its input FHIR resources and evaluation result (met / not met / indeterminate).
  • FHIR Resource Traceability: Each decision is traceable to the underlying FHIR resources (Patient, Observation, Condition, etc.) that informed it.
  • Audit Retention: Audit logs are retained per site agreement and available for institutional compliance reviews.
  • Institutional Access Control: Audit logs are accessible only to authorized institutional personnel and cannot be accessed by Damaros personnel.
04B_PERFORMANCE_AND_USAGE

Performance and Usage Compliance

Damaros adheres to Epic Operational Database (ODB) guidelines and best practices for healthcare integration.

  • Peak Load Compliance: Damaros is designed to consume ≤1% of peak system load on Epic ODB infrastructure.
  • Off-Peak Load Compliance: Off-peak operations are designed to consume ≤5% of available capacity.
  • Bulk Export Preference: Large cohort retrievals (≥1000 patients) use Bulk FHIR Export endpoints to minimize real-time API load.
  • Rate Limiting: All polling and retry logic respects Epic rate limits with exponential backoff (max 5 minutes between retries).
  • Connection Pooling: Persistent connections are reused to minimize handshake overhead.
  • Batch Operations: Search queries use _count parameter and pagination to avoid full-result retrieval.
04C_ERROR_HANDLING_AND_RESILIENCY

Error Handling and Resiliency

Damaros implements comprehensive error recovery to maintain service availability and data integrity.

  • Token Refresh Handling: Expired or revoked tokens trigger automatic re-authentication. No user intervention is required for token refresh.
  • Transient Failure Retry: Network timeouts and transient 5xx errors are retried with exponential backoff (max 3 retries).
  • Partial Failure Handling: If a patient's data retrieval fails, the screening run logs the failure and continues with other patients. Partial failure is reported to the user.
  • Circuit Breaker Pattern: If Epic API errors exceed 10% of requests over a 5-minute window, the system enters graceful degradation mode and alerts the operator.
  • Data Consistency: Idempotent operations ensure that retries do not corrupt audit state or produce duplicate results.
  • Operator Alerting: Critical errors (token endpoint unavailable, database connectivity loss) trigger immediate operator notification.
05_DATA_USE_AND_PRIVACY

Data Use and Privacy

  • Clinical data is used only to evaluate patient eligibility for specific research studies.
  • Data is not sold, licensed, used for advertising, or used to train commercial AI models.
  • Processing is governed by site-specific agreements before activation.
  • Data is encrypted at rest and in transit.
  • Retention follows the applicable site agreement.
