Epic FHIR Integration

Production Integration Documentation

Deployment-facing technical reference aligned to the Damaros product codebase and Epic SMART on FHIR Backend Services (server-to-server). Covers authentication, JWKS publication, authorized reads, PHI boundaries, deterministic screening, access control, and production write-back gates. Not shown here: end-user SMART EHR launch as the primary cohort connector—that path is reference-only in the open-source smart-app stub.

01_PLATFORM_OVERVIEW

Platform Overview

Damaros is a clinical trial execution platform that performs deterministic, protocol-driven eligibility evaluation against Electronic Health Record data.

The system converts trial protocols into structured evaluation logic, screens approved cohorts against inclusion and exclusion criteria, and produces criterion-level PASS, REVIEW, or FAIL outcomes with evidence for coordinator review, institutional oversight, and audit reconstruction.

The production Epic integration path is SMART on FHIR Backend Services: the API/worker exchanges a signed JWT for an access token (client_credentials + private_key_jwt, RFC 7523). Access is read-only by default. Epic access tokens are not issued to browsers and are not stored in the web session.

02B_AUTHENTICATION_FLOW

Authentication Flow (Backend Services)

The API builds a signed JWT (iss/sub = Epic client id, aud = token URL, jti = UUID, bounded exp), POSTs to the Epic token endpoint with grant_type=client_credentials, and caches the access token in process memory for FHIR HTTP clients. Failures surface to operators; there is no silent fallback to unauthenticated reads.

GRANT_TYPEclient_credentials
ASSERTION_TYPEurn:ietf:params:oauth:client-assertion-type:jwt-bearer
TOKEN_ENDPOINTEpic-issued URL (DAMAROS_EPIC_TOKEN_URL)
JWT_ISS / JWT_SUBEpic client ID for the active environment
JWT_AUDToken endpoint URL (per Epic / RFC 7523)
JWT_JTIUUID per assertion
TOKEN_STORAGEIn-process only; not written to disk, DB, or browser
FHIR requests use bounded retry and backoff on 429/5xx per connector configuration.
02C_OAUTH_SCOPE

OAuth Scope

Typical Epic registration uses system-level read scopes (e.g. system/Patient.read, system/Observation.read, …) narrowed to the minimum resource types the site approves. Exact scope strings follow Epic’s vocabulary for your app type. Optional DAMAROS_EPIC_SCOPE passes a space-delimited list to the token request.

Standard submission: no write scopes. Add Bulk Data / Group export scopes only when $export is enabled for that tenant.

04_PHI_BOUNDARY

PHI Boundary and Data Flow

  • Cohorts are defined by approved protocol logic or institutional configuration.
  • Patient data is retrieved through FHIR read APIs or, when enabled, Bulk FHIR export—per site policy.
  • PHI is processed inside the institution-approved deployment boundary.
  • Patient-level clinical data does not enter external LLM API paths; application-layer gates enforce separation (see product compliance documentation).
  • Eligibility is computed deterministically; outputs are criterion-level with evidence references where populated.
  • Audit artifacts preserve source FHIR references where available for reconstruction.
05_AUDIT_AND_REPLAY

Audit, Determinism, and Replay

  • Decisions are reproducible for fixed protocol version, cohort snapshot, evidence snapshot, and evaluation_as_of.
  • Every patient outcome is recorded at criterion level with PASS, FAIL, or REVIEW and rationale.
  • Protocol amendments link to versioned protocol IDs and engine versions.
  • Replay classifies agreement or divergence; no opaque “model decided” eligibility.
05A_ACCESS_CONTROL

Identity and Access Control

  • Role-scoped access (coordinator, investigator, sponsor, site admin, etc.).
  • Principals carry subject, role, organization scope, and trial scope.
  • Administrative and export actions restricted by RBAC; web sessions are TTL-bound.
05B_WRITE_BACK

Write-Back Controls

  • Write-back is off by default in all environments at the code level.
  • Non-production: DAMAROS_FHIR_WRITE_ENABLED=1 after site policy may enable write helpers.
  • Production: requires DAMAROS_FHIR_WRITE_ENABLED=1 and DAMAROS_FHIR_PRODUCTION_WRITE_ATTESTED=1 when DAMAROS_ENV is production-class—prevents a single env typo from enabling EHR writes during IT review.
  • Standard screening does not require Epic write-back.
06_PERFORMANCE

Performance and API Usage

  • FHIR usage follows Epic operational guidance; bulk export preferred for large cohort refresh where approved.
  • Rate limiting and bounded exponential backoff on 429/5xx.
  • Per-resource failures in a bundle do not block other resource types in the same pull.
06A_ERROR_HANDLING

Error Handling and Resiliency

  • Missing clinical data surfaces as REVIEW where appropriate—not silent PASS.
  • Retry paths are idempotent while write-back remains disabled.
07_SECURITY_AND_PRIVACY

Security and Privacy

  • Signing keys and Epic tokens are server-side secrets; TLS for FHIR in transit.
  • Structured logging designed to reduce PHI leakage (deployment-specific log processors).
  • Data at rest encryption and retention are deployment- and agreement-driven.
