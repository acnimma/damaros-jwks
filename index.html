Epic FHIR Integration

Production Integration Documentation

Deployment-facing technical reference for the Damaros Epic SMART on FHIR integration, including authentication, requested services, PHI boundaries, access control, audit guarantees, performance posture, resiliency behavior, and production controls.

01_PLATFORM_OVERVIEW

Platform Overview

Damaros is a clinical trial execution platform that performs deterministic, protocol-driven eligibility evaluation against Electronic Health Record data.

The system converts trial protocols into structured evaluation logic, screens approved cohorts against inclusion and exclusion criteria, and produces criterion-level evidence suitable for coordinator review, institutional oversight, and audit reconstruction.

The Epic integration operates as a SMART on FHIR Backend Services application. Access is read-only by default. Optional write-back capabilities remain disabled unless explicitly enabled after site authorization, IRB/policy review, and deployment-level configuration.

02B_AUTHENTICATION_FLOW

Authentication Flow

Damaros uses SMART on FHIR Backend Services with JWT client assertions. Tokens are requested from the Epic authorization server using client credentials and are scoped to institution-approved system-level access.

GRANT_TYPEclient_credentials
ASSERTION_TYPEJWT bearer client assertion
TOKEN_ENDPOINTEpic-provided authorization server endpoint
JWT_ISS / JWT_SUBEpic-issued client ID for the active environment
JWT_AUDEpic token endpoint URL
JWT_EXPShort-lived assertion, 300 seconds or less
JWT_JTIUnique identifier generated per assertion
TOKEN_STORAGEAccess tokens are held only for the active execution context and are not persisted
Token acquisition is bounded, retry-limited, and visible to operators when authentication fails.

03A_PHI_BOUNDARY_AND_DATA_FLOW

PHI Boundary and Data Flow

  • Cohorts are defined by approved protocol logic or institutional configuration.
  • Patient data is retrieved through FHIR APIs or Bulk FHIR export depending on cohort size and site policy.
  • PHI is processed only inside the institution-approved deployment boundary.
  • Protocol parsing and non-PHI configuration may use separate services, but patient-level PHI is not sent to public LLM APIs.
  • Eligibility is computed deterministically against protocol inclusion and exclusion criteria.
  • Results are surfaced to authorized clinical research users with criterion-level evidence.
  • Audit artifacts preserve source references required for review and reconstruction.
  • No data leaves authorized infrastructure boundaries without institutional approval and executed data-use terms.
04_AUDIT_AND_REPLAY

Audit, Determinism, and Replay

  • Eligibility decisions are deterministic and reproducible for a fixed protocol version, cohort definition, data snapshot, and evaluation_as_of timestamp.
  • Every patient decision is logged at criterion level with PASS, FAIL, or REVIEW status.
  • Evidence links back to source FHIR resources, document metadata, and retrieved payload references where available.
  • Protocol amendments generate versioned re-evaluation outputs and reviewable diffs.
  • Replay uses the same protocol version, evaluation timestamp, normalized evidence set, and deterministic ordering rules.
  • No opaque or black-box decisioning is used for clinical eligibility outputs.
  • Audit logs are institution-scoped, access-controlled, and reviewable by authorized users.
04A_ACCESS_CONTROL

Identity and Access Control

  • User access is role-scoped for clinical research workflows, including coordinator, investigator, administrator, and technical operator roles where enabled.
  • Users see only institution-approved studies, cohorts, evidence, and audit records.
  • Administrative actions, export operations, and write-capable configuration changes are restricted to authorized users.
  • Sessions, authentication, and inactivity controls are configured according to the institution-approved deployment model.
  • Access events are logged for security review and operational audit.
04B_PERFORMANCE_AND_USAGE

Performance and API Usage

  • FHIR API usage is designed to comply with Epic operational performance guidance and site-specific limits.
  • Target usage is bounded to Epic ODB guidance: no more than 1% peak load and no more than 5% off-peak load unless a site approves otherwise.
  • Bulk FHIR export is preferred for large cohort workflows to avoid excessive point-query load.
  • Requests are rate-limited and governed by bounded exponential backoff.
  • Polling intervals are controlled to avoid unnecessary authorization server or FHIR server pressure.
  • Batching, connection reuse, and job-level orchestration reduce system load.
  • Retry behavior is idempotent and avoids duplicate write behavior when optional write-back is disabled.
04C_ERROR_HANDLING

Error Handling and Resiliency

  • Token acquisition failures are handled with bounded retry and operator-visible failure states.
  • Transient FHIR errors use exponential backoff with retry caps.
  • Partial cohort failures are isolated and surfaced for review without corrupting completed results.
  • Missing or stale data is surfaced as REVIEW rather than silently converted into eligibility PASS.
  • Upstream instability can trigger circuit breaker behavior to protect Epic infrastructure.
  • Failed ingestion, screening, export, and replay jobs preserve diagnostic metadata for authorized operators.
  • Operations are designed to be replay-safe under fixed protocol, cohort, and evaluation timestamp inputs.
05_SECURITY_AND_PRIVACY

Security and Privacy

  • Private signing keys are stored in managed secrets infrastructure and never committed to source control.
  • Access tokens are short-lived and not persisted.
  • All access is institution-scoped and bound to approved Epic client credentials.
  • All API traffic is encrypted in transit using TLS 1.2 or higher.
  • Data is encrypted at rest according to deployment and site policy.
  • No PHI is written to application logs at any verbosity level.
  • Clinical data is used only for approved research screening workflows.
  • Data is not sold, licensed, used for advertising, or used to train commercial AI models.
  • Retention, deletion, export handling, and purge behavior follow site-specific Data Use Agreement and institutional policy.
05A_OPTIONAL_WRITE_BACK

Optional Write-Back Controls

  • Write-back is disabled by default in production configuration.
  • Write-back requires explicit site approval, appropriate policy review, and deployment-level enablement.
  • Permitted write-back is limited to institution-approved objects such as consent or screening-status observations.
  • Every write-capable operation is logged with actor, timestamp, patient context, payload class, and result.
  • Standard eligibility screening does not require write-back to Epic.
